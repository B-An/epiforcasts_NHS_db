# Documentation Style Guide

## Audience
All contributors producing documentation, comments, release notes, or operational runbooks.

## Purpose
Ensure documentation is lucid, professional, and consistent in British English.

## Core writing rules

1. Write in UK English.
2. Prefer plain language over dense jargon.
3. Define terms at first use.
4. Keep one key idea per sentence where possible.
5. Distinguish evidence from interpretation.

## Mandatory section template

All substantial pages should include:

1. Audience
2. Purpose
3. Scope
4. Assumptions
5. Limitations
6. Owner
7. References
8. Last updated

## Tone and register

1. Professional and direct.
2. Respectful and neutral.
3. Honest about uncertainty.
4. Never over-claim maturity or safety.

## Ambiguity control

Use explicit status labels:

1. Confirmed
2. Inferred
3. Assumed
4. Unknown

Use them where readers could otherwise infer false certainty.

## Terminology control

1. Use approved terms from [../70-reference/glossary.md](../70-reference/glossary.md).
2. If a term is missing, add it before merge.
3. Avoid synonym drift for core concepts.

## Hyperlink rules

1. Add related links at top or bottom of each page.
2. Link to canonical pages, not temporary notes.
3. Keep links descriptive and stable.

## Numerical and statistical language

1. State units and scale.
2. Differentiate model output from measured quantity.
3. Avoid certainty language unless empirically justified.
4. Explain credible intervals and probabilities in plain English.

## Accessibility and readability

1. Prefer informative headings.
2. Use short tables for comparisons.
3. Keep paragraphs compact.
4. Include a short summary at the start for long pages.

## Visuals

1. Prefer diagrams that clarify flows and responsibilities.
2. Include simple text alternatives nearby.
3. Keep labels plain and audience-appropriate.
